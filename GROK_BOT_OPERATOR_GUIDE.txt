5. WHAT THE BOT DOES (every 5 minutes)
================================================================================

Loop cycle:
  data → signal → verify → execute → monitor → memory → repeat

Stage 1 — Data ingestion
  - Polymarket CLOB: discover btc-updown-5m-* market + orderbook
  - Binance BTCUSDT + Coinbase BTC-USD (leading CEX prices)
  - TradingView webhook signals (leading direction)
  - Chainlink BTC/USD on-chain (settlement truth only)

Stage 2 — Signal generation (Grok MAKER)
  - Computes p_up, edge_bps from leading stack vs window open
  - Grok proposes risk bias/size overlay (never self-grades)

Stage 3 — Verification (CHECKER)
  - Numeric gates: Sharpe > 1.5, drawdown < 10%, Newey-West t > 2.0, OOS >= 2y
  - Claude independently reviews signal — rejects even if numeric passes

Stage 4 — Execution
  - Paper fill only (submit_order with paper=False raises error)

Stage 5 — Risk monitoring
  - Drawdown breaker, kill events logged to STATE.md
  - Self-improvement every 10 windows: Grok proposes lesson → Claude reviews

12. HOW THE LOOP ENGINE WORKS (module-by-module)
================================================================================

Core idea from loop engineering:
  - The LLM (Grok/Claude) FORGETS between runs.
  - The LOOP does NOT — STATE.md, skill files, and JSONL logs survive restarts.
  - You engineer the loop in code; the agents run inside it.

Heartbeat (every cycle):
  data → signal → verify → execute → monitor → memory → repeat

Two runtimes use the same loop/ package:

  A) VPS DAEMON (production) — grok_bot/daemon.py
     Runs forever, one real BTC 5m window every 300 seconds.
     Uses Btc5mWindowEngine (live Polymarket + CEX + Chainlink + TV).
     Also serves dashboard :8800 and TradingView webhook :8799.

  B) DISCOVERY LOOP (batch) — loop/discovery_engine.py
     Runs @goal until edge proven OR window/time budget exhausted.
     Uses factory.build_discovery_engine() with ingest/propose callables.
     Invoked via: python -m grok_bot.main --discover-loop

12.2  AUTOMATION — loop/runner.py
--------------------------------------------------------------------------------

Two trigger types (from the loop engineering blueprint):

  @loop(interval_seconds=N)
    Cadence trigger. Fires on a timer regardless of outcome.
    Used for: per-window paper cycle, per-minute risk checks.
    run_loop(fn) executes fn repeatedly until max_iterations or max_seconds.

  @goal
    Condition trigger. Runs until a CODE-CHECKED condition is True.
    Used for: "accumulate windows until calibration armed OR 5000 windows."
    run_goal(step, condition) calls step(i) each iteration, then checks
    condition() — never trusts an LLM saying "I'm done."

Hard caps on every loop (prevents infinite runaway):
  max_iterations  (default 1,000–10,000)
  max_seconds     (default 86,400 = 24h)

ProfitDiscoveryEngine uses run_goal() with condition = any_armed.

Files:
  loop/skills/alpha_research.md  — how signals/thresholds are chosen
  loop/skills/verification.md  — checker rules (numeric gates)
  loop/skills/risk.md            — kill-switch and drawdown rules

Functions:
  read_skill(name)     — load skill markdown at cycle start
  append_lesson(name, lesson) — append to "## Lessons learned" section

Self-improvement flow writes confirmed lessons here. Example entry:
  - 2026-06-24: Lost on window 42 (DOWN). New rule: raise min_edge_bps
    when leading_confidence < 0.5.

The analyst (Grok) proposes lessons; the reviewer (Claude) must approve
before append_lesson() is called. Threshold changes are staged but only take
effect after re-validation on the next cycle.

--------------------------------------------------------------------------------
12.5  VERIFIER — loop/verifier.py
--------------------------------------------------------------------------------

Independent checker. The maker (Grok) NEVER grades its own work.

verify_signal(signal, rules) → Verdict(passed, reasons, evidence)

Default numeric gates (all code-checked, not LLM claims):
  Sharpe ratio        > 1.5
  Max drawdown        < 10%
  Newey-West t-stat   > 2.0
  Out-of-sample years >= 2.0

Returns Verdict with:
  passed   — True only if ALL gates pass
  reasons  — list of failures (e.g. "sharpe 0.00 < 1.5")
  evidence — numeric values used in the check

Two-layer checking in production (grok_bot/pipeline.py):
  1. verify_signal()     — pure Python, always runs first
  2. ClaudeVerifier      — LLM checker, can veto even if numeric passes

promotion_rung() in grok_bot/evidence.py uses separate portfolio/calibration
gates for the observe → shadow → armed ladder (see section 8).

--------------------------------------------------------------------------------
12.6  FIVE STAGES — loop/driver.py (DiscoveryLoop)
--------------------------------------------------------------------------------

DiscoveryLoop.run_cycle() orchestrates one full pass:

  Stage 1 — INGEST
    Calls ingest() callable → dict with price_stack, edge_bps, p_up, etc.
    On failure: log ingest error, write state, return early.

  Stage 2 — SIGNAL (maker proposes, no self-grading)
    Calls propose(data) → signal dict or None.
    None = no candidate this window (observe only).
    Maker (Grok) may influence proposal via grok_bot/pipeline.py but does
    NOT run verification here.

  Stage 3 — VERIFY (checker)
    verify_signal(signal) → if failed, set last_verdict=rejected, stop.

  Stage 4 — EXECUTE (paper only)
    submit_order(signal, paper=True) → simulated fill.
    paper=False raises LiveTradingDisabledError — structurally impossible.

  Stage 5 — MONITOR
    _drawdown() across open paper positions.
    If drawdown > max_drawdown (5%): clear positions, append_kill_event.
    If verify passed and drawdown OK: current_rung → shadow.

Returns list[CycleResult] — one entry per stage with ok/detail/verdict.

--------------------------------------------------------------------------------
12.7  PROFIT DISCOVERY — loop/discovery_engine.py
--------------------------------------------------------------------------------

ProfitDiscoveryEngine wraps DiscoveryLoop in a @goal search:

  North star: discover whether real edge exists — NOT deploy capital.

  status() — single-screen answer "have we found profit yet?"
    Reads EvidenceStore + StateManager.
    Returns: mode, rung, headline, blockers, portfolio, calibration.

  run() — bounded discovery loop:
    run_goal(step, condition=any_armed)
    Stops when:
      (a) promotion_rung hits "armed"  → stop_reason=edge_proven
      (b) max_windows or max_seconds hit → stop_reason=budget_exhausted
    Notifies operator via loop/connectors/notifier.py on promotion/armed.

  _persist_cycle() — after each window:
    Appends WindowRecord to reports/windows.jsonl
    Updates state.json rung and windows_processed
    Fires promotion notification on rung change

Config (env vars):
  DISCOVERY_MAX_WINDOWS  (default 500, VPS .env may set 5000)
  DISCOVERY_MAX_SECONDS  (default 86400, VPS .env may set 604800)

--------------------------------------------------------------------------------
12.8  SELF-IMPROVEMENT — loop/self_improve.py + grok_bot/analyst.py
--------------------------------------------------------------------------------

"The article's rule: every loss writes a new rule."

Pipeline (maker ≠ checker, dual verification):

  1. Grok ANALYST (grok_bot/analyst.py propose_lesson)
     Receives loss bundle: last loss window, direction, edge, PnL.
     Proposes ProposedChange: lesson text + threshold_key + new_value.

  2. PRE-CHECK (loop/self_improve.py pre_check_change) — pure Python
     - threshold_key not in FROZEN_KEYS (paper_only, live_signoff, min_oos_years)
     - new_value within bounds
     - supporting_signal passes verify_signal()

  3. Claude REVIEWER (grok_bot/analyst.py reviewer_approves)
     Independent LLM grades the proposed change.

  4. APPLY (loop/self_improve.py dual_verify → apply_confirmed_lesson)
     Appends lesson to loop/skills/alpha_research.md.
     Threshold change staged — re-validated next cycle before taking effect.

Daemon triggers self-improvement every 10 windows (if >= 3 trades exist).

12.11  LIVE WINDOW ENGINE — grok_bot/window_engine.py (VPS path)
--------------------------------------------------------------------------------

Btc5mWindowEngine.run_window() — one real 5-minute Polymarket cycle:

  1. PolymarketClient.discover_current() → btc-updown-5m-* market slug
  2. Fetch UP/DOWN orderbook mids from CLOB API
  3. ChainlinkReader.read() → settlement price at window open
  4. WindowIngestor.ingest() → PriceStack (Binance + Coinbase + TV + CL)
  5. build_signal_candidate() → Grok maker overlay on signal dict
  6. verify_signal() → numeric gate
  7. verify_with_checker() → Claude gate
  8. submit_order(paper=True) if UP or DOWN direction confirmed

Returns: {action, data, signal, pnl, market, reasons}

Daemon calls this every 300s, appends to EvidenceStore, updates StateManager,
writes discovery report, runs self-improvement, checks promotion.
